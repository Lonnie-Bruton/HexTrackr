Versioning Strategy

This document outlines the versioning strategy for the HexTrackr application, which follows Semantic Versioning (SemVer) 2.0.0.

Version Format

The version number is defined in the format MAJOR.MINOR.PATCH (e.g., 1.2.3).

  • MAJOR version: Incremented for incompatible API changes or significant architectural shifts.
  • MINOR version: Incremented for new, backward-compatible functionality.
  • PATCH version: Incremented for backward-compatible bug fixes.

Version Management

A dedicated script, scripts/version-manager.js, automates the process of updating the version number across the project.

Key Files Updated

The script ensures that the version number is consistent across the following critical files:

  1. package.json: The canonical source for the project's version.
  2. tickets.html: The version is displayed in the UI footer.
  3. vulnerabilities.html: The version is displayed in the UI footer.

How to Use the Version Manager

The script is run from the command line using Node.js.

1. Check the Current Version

To see the current version of the application, run the script without any arguments:

node scripts/version-manager.js

2. Update to a New Version

To update the application to a new version, provide the new version number as an argument:

node scripts/version-manager.js <new-version>

Example:

node scripts/version-manager.js 1.1.0

The script will then perform the following actions:

  • Validate that the new version string conforms to the X.Y.Z format.
  • Update the version field in package.json.
  • Replace the version number within the <span id="app-version">...</span> tag in both tickets.html and vulnerabilities.html.

Release Process

  1. Create a Backup: Before making changes, ensure your work is committed.
  2. Update Version: Run the version-manager.js script to bump the version number.
  3. Update Changelog: Document the changes in CHANGELOG.md under the new version heading.
  4. Commit Changes: Commit the updated files (package.json, *.html, CHANGELOG.md) with a message like chore: release vX.Y.Z.
  5. Tag the Release: Create a Git tag for the new version.
    git tag vX.Y.Z
git push origin vX.Y.Z
  6. Merge to Main: Merge the changes from the development branch (e.g., copilot) into the main branch for a stable release.

Coding Style Guide

This document outlines the coding style, conventions, and best practices to be followed when contributing to the HexTrackr project. Adhering to these guidelines ensures code consistency, readability, and maintainability.

JavaScript Style Guide

The project uses ESLint to enforce a consistent JavaScript coding style. The configuration is defined in eslint.config.js.

Key ESLint Rules

  • Quotes: Use double quotes (") for all strings.

    • Rationale: Consistency.
    • ESLint Rule: "quotes": ["error", "double"]

  • Semicolons: Always use semicolons at the end of statements.

    • Rationale: Prevents ambiguity and potential issues with Automatic Semicolon Insertion (ASI).
    • ESLint Rule: "semi": ["error", "always"]

  • ECMAScript Version: Code should be compatible with ECMAScript 2022 (ecmaVersion: 2022).

  • Source Type:

    • Backend Node.js files and scripts use CommonJS modules (sourceType: "commonjs").
    • Frontend scripts and modern configuration files use ES Modules (sourceType: "module").

Globals

The .eslintrc.config.js file defines allowed global variables for different parts of the application to prevent accidental use of undefined variables.

  • Node.js Environment: Standard globals like require, module, process, __dirname are permitted in server-side scripts.
  • Browser Environment: Standard browser globals like window, document, fetch, localStorage are permitted in client-side scripts.
  • Libraries: Globals for third-party libraries like bootstrap and Prism are also defined where applicable.

File-Specific Configurations

  • scripts/pages/*.js: Page-specific JavaScript files.
  • scripts/shared/*.js: JavaScript modules shared across multiple pages.
  • docs-prototype/js/*.js: JavaScript for the documentation portal.

HTML Style Guide

  • Indentation: Use two spaces for indentation.
  • Quotes: Use double quotes for all attributes.
  • Semantics: Use semantic HTML5 tags where appropriate (<header>, <footer>, <main>, <section>, <nav>).
  • Accessibility:
    • Provide alt attributes for all images.
    • Use aria-* attributes to improve accessibility for dynamic components.
    • Ensure a logical document outline with proper heading levels (<h1> through <h6>).

CSS Style Guide

  • Naming Convention: Use a consistent naming convention for classes, such as BEM (Block, Element, Modifier) or a similar structured approach.
  • Organization:
    • styles/pages/: Contains page-specific styles.
    • styles/shared/: Contains styles for shared components (e.g., navigation, modals).
    • styles/utils/: Contains utility classes (e.g., spacing, colors).
  • Formatting:
    • Use two spaces for indentation.
    • Place each property-value pair on its own line.
    • Add a space after the colon (:) for each property.

General Best Practices

  • Comments:
    • Write clear and concise comments to explain complex logic.
    • Use JSDoc-style comments for functions to describe their purpose, parameters, and return values.
  • Naming:
    • Use descriptive names for variables, functions, and files.
    • Use camelCase for variables and functions.
    • Use PascalCase for classes.
    • Use kebab-case for filenames.
  • Modularity:
    • Keep files focused on a single responsibility.
    • Break down large functions into smaller, reusable ones.
    • Follow the established modular pattern for JavaScript (scripts/pages/ vs. scripts/shared/).
  • Version Control:
    • Write clear and descriptive Git commit messages.
    • Follow the established Git workflow (work on the copilot branch, create pull requests to main).

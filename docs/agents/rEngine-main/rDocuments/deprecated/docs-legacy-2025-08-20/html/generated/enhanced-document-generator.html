Enhanced Document Generator

Purpose & Overview

The enhanced-document-generator.js script is a powerful tool that automatically generates comprehensive documentation for JavaScript files. It leverages multiple AI-powered APIs (such as OpenAI, Groq, and Gemini) to produce well-structured Markdown documentation, handling challenges like rate limiting and large file sizes.

The key features of this script include:

  • Multi-Provider Support: It can utilize different documentation generation providers, automatically choosing the best one based on available API keys and rate limiting.
  • Rate Limiting Handling: The script incorporates a MultiProviderRateLimiter to track usage and gracefully handle rate limiting across providers.
  • Large File Support: For large code files that exceed the token limits of the providers, the script intelligently splits the content into manageable sections and generates documentation for each.
  • Batch Processing: The script can generate documentation for multiple files in a batch, providing a progress report and error handling.
  • Configuration Management: It reads configuration settings from a system-config.json file and environment variables, making it easy to set up and customize.

This script is designed to streamline the documentation generation process, ensuring that your code is accompanied by clear, comprehensive, and accessible documentation.

Technical Architecture

The EnhancedDocumentGenerator class is the main component of this script, responsible for the following key functionalities:

  1. Configuration Management: It reads the system-config.json file and environment variables to get the necessary configuration settings, such as API endpoints and rate limiting parameters.
  2. Provider Selection: The class determines the available documentation generation providers based on the provided API keys and chooses the best one for each request.
  3. Rate Limiting Handling: It uses the MultiProviderRateLimiter class to track usage and handle rate limiting across the different providers.
  4. Documentation Generation: The class has two main methods for generating documentation:
    • generateDocumentation(filePath): Processes a single file, generating the documentation and saving it to the appropriate output location.
    • generateBatchDocumentation(filePaths): Processes a batch of files, generating documentation for each and providing a consolidated report.
  5. Prompt Building: The buildDocumentationPrompt() method constructs the prompt used to generate the documentation for a given code file.
  6. File Splitting: For large files that exceed the token limits of the providers, the splitContentIntoSections() method splits the content into manageable sections and processes each section individually.

The script also includes a command-line interface (CLI) that allows users to generate documentation for a single file, a batch of files, or view the current provider status.

Dependencies

The enhanced-document-generator.js script relies on the following external dependencies:

  • fs-extra: Provides an enhanced file system API with additional functionality.
  • path: Handles file and directory paths.
  • axios: Makes HTTP requests to the documentation generation providers.
  • url: Provides utilities for URL resolution and parsing.
  • dotenv: Loads environment variables from a .env file.
  • multi-provider-rate-limiter.js: A custom module that handles rate limiting across multiple providers.

Usage Examples

Generating Documentation for a Single File

node enhanced-document-generator.js file js/inventory.js

This command will generate the documentation for the inventory.js file and save it to the docs/generated directory.

Generating Documentation for a Batch of Files

node enhanced-document-generator.js batch "js/*.js"

This command will generate the documentation for all JavaScript files in the js directory and save the results to the docs/generated directory. It will also create a batch-documentation-report.json file in the docs directory, containing a summary of the generation process.

Checking Provider Status

node enhanced-document-generator.js status

This command will display the current status of the available documentation generation providers, including their usage statistics and rate limiting information.

Configuration

The enhanced-document-generator.js script reads its configuration from two sources:

  1. system-config.json: This file, located in the same directory as the script, contains the following configuration settings:

    • brainShareSystem.documentManager.api: An object that defines the API endpoints, provider-specific settings, and fallback order.

  2. Environment Variables:

    • GROQ_API_KEY: The API key for the Groq provider.
    • OPENAI_API_KEY: The API key for the OpenAI provider.
    • GEMINI_API_KEY: The API key for the Gemini provider.

Make sure to create the system-config.json file and provide the necessary API keys as environment variables for the script to function correctly.

Error Handling

The EnhancedDocumentGenerator class handles several types of errors:

  1. File Not Found: If the specified file does not exist, the script will log an error and skip the file.
  2. Empty File: If the file is empty, the script will log a warning and skip the file.
  3. Provider Configuration Issues: If a provider is not configured correctly (e.g., missing API key), the script will log an error and try the next available provider.
  4. Provider Request Failures: If a provider request fails, the script will log the error and attempt to use the next available provider. If all providers fail, the script will log a final error.
  5. Large File Handling Errors: If a section of a large file fails to generate, the script will log the error and include a placeholder comment in the generated documentation.

The script also provides a consolidated batch report that includes the success/error counts and details for each file processed.

Integration

The enhanced-document-generator.js script is designed to be integrated into a larger system or workflow. Some possible integration points include:

  1. Build Automation: The script can be included in the build process of a project, automatically generating up-to-date documentation for the codebase.
  2. Documentation Management: The generated documentation can be integrated into a documentation management system or hosted on a website.
  3. Continuous Integration/Deployment: The script can be part of a CI/CD pipeline, ensuring that documentation is always in sync with the codebase.
  4. Developer Tooling: The script can be used as a standalone tool or integrated into IDEs or other development environments to provide on-demand documentation generation.

Development Notes

  1. Multi-Provider Flexibility: The script's design allows for easy integration of new documentation generation providers. To add a new provider, you would need to implement the necessary request handling logic in the makeDocumentRequest() method.
  2. Token Estimation: The script uses a simple character-based token estimation method. For more accurate token counting, you may need to integrate a dedicated token counting library or service.
  3. Batch Processing Optimization: The current batch processing implementation is a basic loop-based approach. For larger batches, you may want to consider using a more efficient parallelization strategy, such as with worker threads or a job queue.
  4. Configurable Output: The script currently saves the generated documentation to a fixed location (docs/generated). You may want to make the output location configurable or provide options to customize the output format and structure.
  5. Comprehensive Error Handling: While the script handles various errors, you may want to expand the error handling and reporting mechanisms to provide more detailed and actionable information to users.
  6. Compatibility with Modern Node.js: The script uses ECMAScript modules (ES6 imports/exports) and Node.js features like fileURLToPath(). If you need to support older Node.js versions, you may need to transpile the code or use a different module system.
